ChatTime 仓库导读

这是一份给“组会前头有点晕的人”准备的仓库说明。你可以把它理解成一张地图：这个项目想做什么、数据怎么流、模型怎么训、奇怪的库分别在干嘛、你汇报时应该先讲哪几件事。

主题：时间序列 + 大语言模型 核心骨架：PyTorch + Transformers 训练方式：LoRA / 4-bit 微调 关键技巧：把时间序列写成“文本”

一句话理解仓库

ChatTime 的核心想法：先把连续时间序列做缩放、离散化、序列化，变成类似文本 token 的形式，再交给 Llama 这类语言模型去做预测或问答。

原始数值序列
例如一段传感器、交通、功率等历史观测值。
离散化与序列化
把连续值映射到固定“词表”，再拼成字符串，如 ###0.1234###。
拼成 Prompt
使用统一模板，把“任务说明 + 输入序列 + 可选上下文”交给 Llama。
文本生成
模型生成后续 token，相当于生成未来时间点的离散值。
反序列化与反离散化
把生成文本还原回数值预测结果。

你在组会里最该先讲什么

  • 1. 这不是传统时序模型 它不是单纯用 Transformer Encoder 直接吃数值，而是把时序当成一种“外语”给 LLM 读。
  • 2. 核心模型不是从零训练 基座来自 Llama，然后通过连续预训练和指令微调，适配时间序列任务。
  • 3. 仓库最近新增了工程化链路 包括本地数据集读取、DAM 数据构建、自动微调和自动评估。
  • 4. 目前多变量能力是“折中版” 一个目标序列做主预测，其余通道主要作为文本上下文注入，而不是严格联合编码。

仓库结构

  • model/model.py 推理主入口。定义 ChatTime 类，负责加载模型、构造生成流程、把文本输出还原成预测值。
  • utils/tools.py 数值预处理核心。包括 Discretizer 和 Serializer。
  • utils/prompt.py Prompt 模板库。把预测任务、分析任务包装成统一文本格式。
  • training/pretrain.py 连续预训练脚本。基于基础 Llama 扩词表、加 LoRA、训练时序语言建模能力。
  • training/finetune.py 指令微调脚本。对 ChatTime 数据集或本地数据做监督微调。
  • training/build_dam_finetune_dataset.py 把 DAM CSV 构造成 ChatTime 可训练的 jsonl 样本。
  • training/evaluate_dam_model.py 加载模型逐条预测，输出 MSE / MAE / RMSE。
  • training/*.sh 实际运行脚本，方便一键预训练、微调和 DAM 任务实验。
  • hf.py 从 Hugging Face 下载模型权重的简化脚本。

阅读顺序

  • 第 1 步：先看 README.md 建立这个项目到底解决什么问题的总体印象。
  • 第 2 步：看 utils/tools.py 这是最关键的“时序转文本”入口，不懂这个就会一直觉得仓库很玄。
  • 第 3 步：看 utils/prompt.py 理解模型最后吃进去的并不是纯数字，而是 instruction-style 文本。
  • 第 4 步：看 model/model.py 你会明白推理时如何反复生成未来片段、再拼成完整预测。
  • 第 5 步：看 training/finetune.py 和 training/pretrain.py 这时再看训练脚本，就不会被一堆库名吓到。
  • 第 6 步：最后看 DAM 扩展脚本 它们是“面向具体项目的工程封装”，不是论文核心算法本身。

仓库真正的主线

阶段 做什么 对应文件 为什么重要
数值转 token 缩放数值，映射到离散中心，再编码成文本字符串 utils/tools.py 这是 ChatTime 和普通 Llama 项目最大的不同
prompt 构造 把预测任务、分析任务写成 instruction / input / response 格式 utils/prompt.py 让时序任务适配 LLM 的输入输出形式
模型推理 加载 Llama，生成文本，再还原成未来数值 model/model.py 把“语言模型输出”变回“时间序列预测”
连续预训练 扩充时序词表，在时序语料上继续训练基础模型 training/pretrain.py 让基座模型学会读这种特殊 token
指令微调 对预测/问答样本做监督微调 training/finetune.py 让模型更像一个能执行时序任务的助手
项目特化 把 DAM 业务数据打包成 ChatTime 格式并做评估 training/build_dam_finetune_dataset.py 等 把论文方法接到真实数据上

那些“奇怪的库”到底是干嘛的

  • torch 底层引擎。张量计算、GPU、模型执行、训练都靠它。没有它，整个仓库转不起来。
  • transformers Hugging Face 大模型工具箱。这里主要负责 Llama 模型、tokenizer 和生成接口。
  • datasets Hugging Face 数据集库。负责读 json/jsonl/csv/parquet、本地磁盘数据集和远程数据集。
  • unsloth 省显存、提速的大模型微调封装。这个库会让 LoRA 训练更省资源，所以你会在训练脚本里频繁看到它。
  • trl 训练器库。这里主要用了 SFTTrainer，相当于现成的监督微调训练循环。
  • peft 参数高效微调库。LoRA 本质上属于它这类方法。代码没有直接 import 很多，但训练逻辑依赖它的生态能力。
  • accelerate 训练和设备调度的底层胶水层，很多 Hugging Face 训练工具会间接依赖它。
  • numpy / pandas 普通数值处理和表格处理，不神秘。主要负责时间序列数组和 CSV 数据。
  • scikit-learn 这里只是借用了 MinMaxScaler 做缩放，不是在搞传统机器学习建模。
  • huggingface-hub 下载模型和数据集用，比如 hf.py 就在做这件事。
  • sentencepiece / safetensors 这类更多是 Llama / HF 生态配套依赖，一个是分词相关，一个是权重文件格式相关。

最关键的 3 个自定义模块

Discretizer 先把连续数值缩放到固定范围，再映射到预定义离散中心。相当于把“连续值”变成“词表里的词”。
Serializer 把离散后的值写成字符串，比如 ###0.1234###。这样 LLM 就能按文本来读。
getPrompt 把任务包装成固定模板：系统说明、指令、输入、输出。预测和分析走的是不同模板。
如果你只能记住一个点，就记住这个：ChatTime 不是直接让 Llama 看原始浮点数，而是先把时间序列翻译成一种特殊文本语言。

推理流程

hist_data -> Discretizer.discretize(...) -> Serializer.serialize(...) -> getPrompt(flag="prediction", ...) -> transformers.pipeline("text-generation") -> 解析 generated_text -> inverse_serialize(...) -> inverse_discretize(...) -> 得到预测值

为什么要分段预测

在 model/model.py 里，模型不是一次性永远生成所有未来点，而是按 max_pred_len 分段生成。生成一小段后，再把预测结果接回历史序列，继续往后推。这是为了更稳地处理较长预测区间。

训练流程

Llama 基座 -> 加载 tokenizer -> 视情况扩充时序词表 -> FastLanguageModel.from_pretrained(...) -> get_peft_model(...) 加 LoRA -> datasets.load_dataset(...) -> TRL 的 SFTTrainer -> TrainingArguments -> trainer.train() -> 保存合并后的模型

为什么会出现 LoRA / 4-bit

因为 7B 级别模型直接全量训练太吃显存，所以这里只训练较少的增量参数，并且可选 4-bit 加载来降低资源开销。这是大模型微调里非常常见的工程做法。

DAM 扩展是怎么接进来的

仓库里新增的 DAM 相关脚本，本质是在做一件事：把业务 CSV 转成 ChatTime 原本就能训练的 prompt 样本，而不是重写整个模型结构。

脚本 作用 你可以怎么理解
training/build_dam_finetune_dataset.py 扫描 CSV 中的目标列，切历史窗口和未来窗口，拼上下文文本，输出 train/validation/test jsonl 把原始业务数据翻译成 ChatTime 能吃的监督样本
training/finetune_dam_1h.sh 如果数据不存在先构造，再启动微调，最后自动评估 一键实验入口
training/evaluate_dam_model.py 逐条跑预测并统计误差 实验结果落盘，不只是看训练日志
这个 DAM 方案目前不是“真正多变量联合时序建模”。更准确地说，它是单目标序列预测 + 其他通道文本上下文辅助。

你汇报时可以怎么描述方法

方法定位 ChatTime 把时间序列映射成离散 token 序列，用大语言模型统一处理预测和问答任务。
训练路径 先做连续预训练，让模型适应时序 token；再做指令微调，让模型学会按任务要求输出预测或分析结果。
工程实现 基于 Hugging Face + PyTorch 生态，使用 LoRA 和 4-bit 技术降低训练成本。
当前扩展 新增了本地数据集兼容、DAM 数据构造、自动微调与评测流程。

最容易卡住的点

  • “为什么 tokenizer 要加新 token？” 因为时间序列被编码成特殊数字串，原始 Llama 词表不一定最适配这种形式。
  • “为什么用语言模型做预测？” 因为作者希望统一处理数值序列和文本上下文，并复用 LLM 的泛化能力。
  • “为什么代码里看起来像 NLP 项目？” 因为它本质上就是把时序任务翻译成 NLP 形式来做。
  • “为什么有 sklearn？” 只是拿来做缩放，不是这个仓库的建模核心。

10 分钟速记版

1. ChatTime = 把时间序列离散化成 token，再交给 Llama。 2. 核心不是“新模型结构”，而是“时序转文本”的表示方式。 3. 推理入口在 model/model.py。 4. 核心魔法在 utils/tools.py 和 utils/prompt.py。 5. 训练主要靠 transformers + datasets + trl + unsloth。 6. LoRA / 4-bit 是为了省显存，不是新算法本体。 7. DAM 扩展是把业务 CSV 变成 ChatTime prompt 数据。 8. 当前 DAM 方案是单目标预测 + 其他特征文本上下文，不是严格多变量联合建模。

如果你只剩最后 3 分钟

优先记这三句话：

  • 第一句 这个仓库的核心不是重新设计一个时序神经网络，而是把时间序列编码成文本 token，让 Llama 来做预测和分析。
  • 第二句 工程上它主要依赖 Hugging Face 生态和 PyTorch，陌生库大多只是大模型微调工具链，比如 Unsloth、TRL、PEFT。
  • 第三句 当前对 DAM 数据的适配方式是保留原有 ChatTime 架构，把目标序列作为主输入，其他变量作为文本上下文辅助。
